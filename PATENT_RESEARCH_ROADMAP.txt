Patent Explorer Roadmap
Last updated: 2026-03-22

Goal
Evolve the app from basic search and project organization into a practical patent research workspace for prior-art, competitive intelligence, and ongoing monitoring.

Current Baseline (already in app)
- Patent search across providers (USPTO, EPO, WIPO, Google Patents).
- Save searches to projects.
- Pin patents to projects/collections.
- Project tabs: Searches, Patents, Notes, Team.
- Search comparison flow is wired.
- Project settings modal supports metadata update/archive/delete.

Guiding Priorities
1. Research precision: better query control and filtering.
2. Review throughput: reduce analyst time to identify relevant art.
3. Defensible output: make results exportable and auditable.
4. Monitoring: support recurring research workflows.

PHASE 1 (Quick Wins, 1-2 sprints)

1) Advanced Filters + Facets (highest ROI)
Why
- Researchers need to narrow noisy result sets fast.

Scope
- Add filters to search UI:
  - Filing date range
  - Grant date range
  - Assignee (contains)
  - Inventor (contains)
  - Provider/jurisdiction multi-select
- Add facet summary in results:
  - Top assignees
  - Filing year histogram
  - Provider split

Implementation notes
- Extend search input model in SearchBar and Index state.
- Pass filters to searchPatents/searchAllProviders.
- Update patentApi mock filtering logic for deterministic behavior.
- Add filter chips above table for active filters.

Suggested files
- src/components/SearchBar.tsx
- src/pages/Index.tsx
- src/lib/patentApi.ts
- src/components/StatsBar.tsx
- src/components/PatentTable.tsx

Acceptance criteria
- User can apply and clear each filter independently.
- Result count and table update correctly for combined filters.
- At least 4 tests for filter behavior (single + combined filters).

done --

2) Query Builder (Boolean + fielded)
Why
- Precision searching is mandatory for real prior-art work.

Scope
- Support syntax examples:
  - "solid state battery" AND assignee:toyota
  - (graphene OR silicon) AND title:electrode
  - cpc:H01M NOT abstract:separator
- Add query helper UI with examples and validation hints.

Implementation notes
- Add lightweight parser/validator (client-side) first.
- Convert parsed terms to patentApi filtering logic.
- Keep raw query string for auditability.

Suggested files
- src/components/SearchBar.tsx
- src/lib/patentApi.ts
- src/lib/patentApi.test.ts

Acceptance criteria
- Invalid syntax gives clear inline message.
- Fielded terms and Boolean operators are honored in result filtering.
- Query string and parsed form are both persisted with saved search.

3) Review Workflow States
Why
- Teams need triage status, not just pinned/not pinned.

Scope
- Add patent review statuses per project:
  - TO_REVIEW, RELEVANT, KEY_PRIOR_ART, EXCLUDED
- Bulk actions in Patents tab.
- Optional rationale note per status change.

Implementation notes
- Extend PatentReference type to include status and statusReason.
- Add update APIs in projectRepository.
- Add filter tabs in Patents tab by status.

Suggested files
- src/types/projects.ts
- src/lib/projectRepository.ts
- src/components/projects/tabs/PatentsTab.tsx
- src/lib/projectRepository.test.ts

Acceptance criteria
- Status can be updated and persists after reload.
- Bulk update works for multi-select.
- Status counts visible in UI.
